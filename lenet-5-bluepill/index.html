    SD-Card–Backed LeNet-5

    SD-Card-Backed LeNet-5 Inference on STM32 Blue Pill

    This example evaluates a LeNet-5-style inference pipeline on an STM32F103 Blue Pill. The main goal is to show how Noodle can execute a model whose full parameter set cannot fit into the microcontroller’s internal memory. The STM32F103 Blue Pill is a useful target because it is highly constrained:

    Flash : 64 KB nominal on common STM32F103C8 boards
SRAM  : 20 KB

    A float32 LeNet-style model quickly exceeds these limits. Therefore, this implementation stores the model weights and biases on an external SD card, while still using RAM for small activation buffers whenever possible.

    In this implementation:

    • weights and biases are stored as raw .bin files on the SD card,
    • large intermediate convolution outputs are stored as files,
    • small fully connected activation vectors are kept in RAM,
    • and the network is executed layer by layer using Noodle.

    The use of .bin files is important. Text files are easy to inspect, but they require ASCII parsing on the microcontroller. Binary files store raw float32 values directly, making them more suitable for repeated embedded inference.

    Why Use an SD Card?

    The SD card is not used merely as a convenience. It is required because the full float32 parameter set is too large to store as const arrays in the flash memory of a typical Blue Pill. For the LeNet-style network used here, the parameter sizes are approximately:

    Layer Parameter shape Number of floats Size
    Conv1 weights 6 × 1 × 5 × 5 150 600 B
    Conv1 bias 6 6 24 B
    Conv2 weights 16 × 6 × 5 × 5 2,400 9.6 KB
    Conv2 bias 16 16 64 B
    FC1 weights 400 × 120 48,000 192 KB
    FC1 bias 120 120 480 B
    FC2 weights 120 × 84 10,080 40.3 KB
    FC2 bias 84 84 336 B
    FC3 weights 84 × 10 840 3.36 KB
    FC3 bias 10 10 40 B

    The total parameter count is:

    61,706 float32 values

    At 4 bytes per float:

    61,706 × 4 = 246,824 bytes ≈ 241 KB

    This is only the model parameter size. It does not include:

    • program code,
    • the Noodle library,
    • the SdFat library,
    • serial communication code,
    • stack,
    • heap,
    • or runtime buffers.

    Therefore, storing all weights as const flash arrays is not feasible on a typical STM32F103C8 Blue Pill. Even a 128 KB variant would not comfortably hold the full float32 model together with the firmware. The SD card solves this by acting as external parameter storage.

    Storage Strategy

    The main idea is to use hybrid memory placement. Large tensors that cannot fit in flash or SRAM are stored on the SD card. Small tensors that fit comfortably in SRAM are kept in RAM to avoid unnecessary file traffic. Noodle supports several execution patterns:

    Pattern Meaning
    RAM → RAM Fastest path, used when both input and output fit in SRAM
    RAM → file Input is in memory, output is written to SD card
    file → RAM Input is loaded from SD card, output is kept in memory
    file → file Both input and output are stored externally

    The important result from the Blue Pill experiment is that not every tensor should be stored as a file. The weights must stay on the SD card because they are too large, but small activation vectors should remain in RAM. The fully connected activation sizes are small:

    FC1 input  : 400 floats = 1600 B
FC1 output : 120 floats = 480 B
FC2 output : 84 floats  = 336 B
FC3 output : 10 floats  = 40 B

    These are small enough for the 20 KB SRAM. Therefore, the optimized version keeps the fully connected activations in RAM, while still reading the fully connected weights from SD card files.

    Layer-by-Layer Data Flow

    The optimized Blue Pill implementation uses the following data placement.

    Step Operation Input Output Parameters Notes
    1 Conv1 + Pool GRID in RAM out1.bin on SD card w01.bin, b01.bin on SD card First convolution starts from the received 28×28 image
    2 Conv2 + Pool out1.bin on SD card out2.bin on SD card w02.bin, b02.bin on SD card File-backed convolution stage
    3 Flatten out2.bin on SD card BUFFER1 in RAM — Produces a 400-float vector
    4 FC1 BUFFER1 in RAM BUFFER2 in RAM w03.bin, b03.bin on SD card Weights remain on SD, activation output stays in RAM
    5 FC2 BUFFER2 in RAM BUFFER1 in RAM w04.bin, b04.bin on SD card Avoids file-backed activation input/output
    6 FC3 + Softmax BUFFER1 in RAM BUFFER2 in RAM w05.bin, b05.bin on SD card Final 10-class output remains in RAM

    This placement is much faster than storing every fully connected activation as a file.

    Noodle Implementation Sketch

    The optimized Noodle call sequence is:

    V = noodle_conv_float(GRID, 1, 6,
                      "out1.bin", 28,
                      cnn1, pool, NULL);

V = noodle_conv_float("out1.bin", 6, 16,
                      "out2.bin", V,
                      cnn2, pool, NULL);

V = noodle_flat("out2.bin", BUFFER1, V, 16);

// FC activations stay in RAM.
// FC weights are still read from SD card.

V = noodle_fcn(BUFFER1, V, 120,
               BUFFER2, fcn1, NULL);

V = noodle_fcn(BUFFER2, V, 84,
               BUFFER1, fcn2, NULL);

V = noodle_fcn(BUFFER1, V, 10,
               BUFFER2, fcn3, NULL);

    The complete flow is:

    GRID
  ↓ Conv1 + Pool
out1.bin
  ↓ Conv2 + Pool
out2.bin
  ↓ Flatten
BUFFER1
  ↓ FC1
BUFFER2
  ↓ FC2
BUFFER1
  ↓ FC3 + Softmax
BUFFER2

    The important change is that out3.bin and out4.bin are no longer used for the fully connected activations. Those vectors are small enough to stay in SRAM.

    SD Card Interface

    The Blue Pill inference node uses an SD card connected through SPI2:

    SD CLK   → PB13
SD MISO  → PB14
SD MOSI  → PB15
SD CS    → PB12
SD 3V3   → 3.3V
SD GND   → GND

    In the tested PlatformIO STM32duino setup, the SD card is initialized with an explicit SPI2 object:

    static const uint8_t SD_CS = PB12;

SPIClass SPI_SD(PB15, PB14, PB13, PB12);  // MOSI, MISO, SCK, SS

if (!noodle_fs_init(SD_CS, SPI_SD, 12)) {
  Serial.println(F("noodle_fs_init FAILED"));
  while (true) delay(1000);
}

    The tested setup was stable at 12 MHz SPI. Increasing the SPI clock from 8 MHz to 12 MHz reduced inference time significantly, indicating that the SD-backed path is strongly affected by SPI transfer speed. However, 18 MHz was not stable with the tested wiring/module, so 12 MHz was used for the reported result.

    Benchmark Result

    Two memory-placement strategies were tested.

    Fully file-backed FC activations

    In the first version, the fully connected activations were also written to and read from SD card files:

    V = noodle_fcn(BUFFER1, V, 120, "out3.bin", fcn1, NULL);
V = noodle_fcn("out3.bin", V, 84, "out4.bin", fcn2, NULL);
V = noodle_fcn("out4.bin", V, 10, BUFFER2, fcn3, NULL);

    This worked, but it was slow:

    Mean inference time ≈ 39.7 s

    The dominant layer was FC2, which took about 33 s. This showed that file-backed activation handling inside the fully connected layers created excessive SD traffic.

    RAM-backed FC activations

    In the optimized version, only the large weights remain on SD card. The small FC activation vectors stay in RAM:

    V = noodle_fcn(BUFFER1, V, 120, BUFFER2, fcn1, NULL);
V = noodle_fcn(BUFFER2, V, 84, BUFFER1, fcn2, NULL);
V = noodle_fcn(BUFFER1, V, 10, BUFFER2, fcn3, NULL);

    This reduced inference time to approximately:

    Mean inference time ≈ 3.47 s

    Representative layer timings were:

    Layer Time
    Conv1 + Pool ≈ 0.424 s
    Conv2 + Pool ≈ 1.47 s
    Flatten ≈ 0.015 s
    FC1 ≈ 1.08 s
    FC2 ≈ 0.43 s
    FC3 + Softmax ≈ 0.056 s
    Total ≈ 3.47 s

    This is about an 11× speedup compared with the fully file-backed FC activation version.

    Why This Matters

    This example demonstrates the central idea behind Noodle:

    Large tensors that do not fit on-chip should be stored externally.
Small tensors that fit in SRAM should stay in SRAM.

    For this Blue Pill example:

    Weights and biases → SD card
Convolution outputs → SD card
Flattened vector and FC activations → RAM
Final softmax output → RAM

    The SD card makes it possible to run a full float32 LeNet-style model whose parameters alone require about 241 KB. The Blue Pill cannot store those parameters as const arrays in flash, and it cannot load them fully into SRAM. However, by streaming weights from SD and carefully keeping only selected activations in RAM, Noodle can still execute the model.

    This is different from a conventional in-memory inference engine. In TensorFlow Lite Micro, the model is normally stored as a memory-addressable flatbuffer, often compiled into flash as a C array, and the tensor arena must fit in SRAM. That works well when the quantized model and arena fit on-chip. In contrast, this Noodle example targets the case where the model parameters exceed the internal flash capacity of the microcontroller.

    Why Not Just Quantize?

    Quantization is an important and often necessary optimization for TinyML deployment. If the model can be quantized to int8 and the resulting model plus tensor arena fits into on-chip flash and SRAM, then a conventional in-memory inference engine such as TFLite Micro is usually the better choice for speed. Quantization would reduce the parameter set from about 241 KB to about 61 KB. So on a nominal 64 KB flash STM32F103C8, even int8 LeNet may still be tight or impractical as a fully compiled-in model. On a 128 KB board, int8 becomes more realistic.

    However, this example explores a different constraint: running a float32 LeNet-style model whose parameter set alone is larger than the internal flash of the target MCU. In this situation, the SD card is not used to compete with quantization, but to provide external parameter storage. Noodle makes the memory placement explicit: large weights are streamed from SD, while small activation vectors are kept in RAM. Thus, quantization and SD-backed execution are complementary. Quantization reduces tensor size; Noodle controls tensor placement.

    Summary

    This example demonstrates SD-card-backed LeNet-style inference on an STM32F103 Blue Pill.

    The main findings are:

    • the complete float32 parameter set is about 241 KB,
    • this is too large for the internal flash of a typical Blue Pill,
    • SD card storage allows the full model weights to be used,
    • fully file-backed activation placement is correct but slow,
    • keeping small FC activations in RAM reduces inference time from about 39.7 s to about 3.47 s,
    • and the final design uses SD for large weights while preserving SRAM for small, high-reuse activations.

    In short, this example shows that Noodle can run a model that exceeds the on-chip memory capacity of the microcontroller by explicitly controlling where each tensor lives.
