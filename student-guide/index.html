    Student Guide: Image Denoising with Noodle

    COLAB NOTEBOOK

    1. Goal

    In this exercise, students deploy a small convolutional autoencoder on a microcontroller using Noodle. The system receives a noisy 28 × 28 grayscale image from a Python program, performs denoising inference on the microcontroller, and sends the reconstructed image back to the computer.

    The complete workflow is:

    1. Train an autoencoder in Python/Keras.
    2. Export the trained weights into Noodle-compatible C header files.
    3. Compile and upload the Arduino/ESP32 firmware.
    4. Send a noisy image over serial.
    5. Receive and visualize the denoised output.

    2. Model Used in This Case

    The example autoencoder uses this structure:

    Stage Layer Channels Kernel Stride Padding Activation Output
    Input Image 1 - - - - 28 × 28 × 1
    Encoder 1 Conv2D 1 → 16 3 × 3 2 same ReLU 14 × 14 × 16
    Encoder 2 Conv2D 16 → 32 3 × 3 2 same ReLU 7 × 7 × 32
    Decoder 1 Conv2DTranspose 32 → 16 3 × 3 2 same ReLU 14 × 14 × 16
    Decoder 2 Conv2DTranspose 16 → 1 3 × 3 2 same Sigmoid 28 × 28 × 1

    In the microcontroller firmware, Noodle stores tensors in CHW layout:

    X  : 1  × 28 × 28
Z1 : 16 × 14 × 14
Z2 : 32 × 7  × 7
Z3 : 16 × 14 × 14
Y  : 1  × 28 × 28

    3. Padding: SAME and VALID in Noodle

    Padding controls what happens near the border of an image or feature map during convolution.

    In Keras, students usually see:

    padding="same"
padding="valid"

    In Noodle, padding is controlled by the P field inside the ConvMem structure:

    ConvMem E1;
E1.K = 3;
E1.P = 65535;
E1.S = 2;
E1.OP = 0;

    For this guide, the important convention is:

    P = 65535  means SAME padding
P = 0      means VALID padding
P = n      means explicit padding of n pixels

    3.1 VALID padding

    VALID padding means no zero padding is added around the input.

    The convolution kernel is only applied where it fully fits inside the input.

    For normal Conv2D, the output width is:

    V = floor((W - K + 2P) / S) + 1

    For VALID padding:

    P = 0

    so:

    V = floor((W - K) / S) + 1

    Example:

    Input width W = 28
Kernel K = 3
Stride S = 1
Padding P = 0

Output width = floor((28 - 3) / 1) + 1
             = 26

    So a 28 × 28 input becomes 26 × 26.

    In Noodle:

    conv.P = 0;

    means VALID padding.

    3.2 SAME padding

    SAME padding means Noodle adds virtual zero padding around the input so that the output width follows the common deep-learning behavior.

    For stride 1, SAME usually keeps the spatial size:

    28 × 28 → 28 × 28

    For stride 2, SAME reduces the size approximately by half:

    28 × 28 → 14 × 14
14 × 14 → 7 × 7

    This is what the image denoising autoencoder uses:

    enc1: 28 × 28, stride 2, SAME → 14 × 14
enc2: 14 × 14, stride 2, SAME → 7 × 7

    In Noodle, use:

    conv.P = 65535;

    to request SAME padding.

    Why 65535?

    Because P is stored as an unsigned 16-bit value. The value 65535 is used as a special marker, not as a real padding size. It tells Noodle:

    compute SAME padding automatically

    So this:

    E1.K = 3;
E1.P = 65535;
E1.S = 2;

    means:

    3 × 3 convolution
SAME padding
stride 2

    3.3 Explicit padding

    Noodle can also use an explicit padding value.

    Example:

    conv.P = 1;

    means one zero-padding pixel is added around the input.

    For a 3 × 3 kernel with stride 1, this behaves like SAME padding:

    W = 28
K = 3
S = 1
P = 1

V = floor((28 - 3 + 2×1) / 1) + 1
  = 28

    However, for stride 2, SAME padding is more subtle, especially when the input size is odd or when output-size rounding matters. Therefore, for Keras-style SAME behavior, students should use:

    conv.P = 65535;

    not manual guessing.

    3.4 Padding in Conv2DTranspose

    The decoder uses Conv2DTranspose:

    D1.OP = 1;
D2.OP = 1;

    and also uses:

    D1.P = 65535;
D2.P = 65535;

    For this denoising model, that means the transposed convolution should mirror the encoder shape:

    dec1: 7 × 7,  stride 2, SAME → 14 × 14
dec2: 14 × 14, stride 2, SAME → 28 × 28

    So the full spatial flow is:

    28 → 14 → 7 → 14 → 28

    This is why W_final should be 28:

    if (W_final != W_OUT) {
  Serial.printf("ERR_BAD_W %u\n", W_final);
}

    If the padding settings are wrong, the final width may not return to 28.

    3.5 Padding used in this image denoising example

    All four layers use SAME padding:

    E1.P = 65535;
E2.P = 65535;
D1.P = 65535;
D2.P = 65535;

    The intended shape sequence is:

    Layer Operation Padding Input width Output width
    E1 Conv2D SAME 28 14
    E2 Conv2D SAME 14 7
    D1 Conv2DTranspose SAME 7 14
    D2 Conv2DTranspose SAME 14 28

    For students, the practical rule is:

    Use P = 65535 when the Keras model says padding="same".
Use P = 0 when the Keras model says padding="valid".
Use P = n only when you intentionally want explicit n-pixel padding.

    Place these files in one project folder:

    project/
├── main.cpp
├── noodle.h
├── noodle.cpp
├── noodle_serial.h
├── noodle_serial.cpp
├── model_exporter.py
├── Auto_Encoder.ipynb
├── w01.h
├── b01.h
├── w02.h
├── b02.h
├── w03.h
├── b03.h
├── w04.h
└── b04.h

    The wXX.h and bXX.h files are generated after training and exporting the model.

    Important note about older main.cpp examples

    Some older Noodle examples allocate one permanent tensor for every layer output, such as X, Z1, Z2, Z3, and Y. Some older examples also do not allocate and register the Noodle temporary buffer.

    For this guide, students should use the safer version:

    BUF_A + BUF_B = ping-pong tensor storage
TEMP2         = Noodle Conv2D scratch storage

    Do not copy an old main.cpp directly unless it has been updated to allocate:

    BUF_A
BUF_B
TEMP2

    and call:

    noodle_setup_temp_buffers(TEMP2);

    Without TEMP2, the memory-to-memory Conv2D path may return 0, and the firmware may later print:

    ERR_BAD_W 0

    5. Step 1 — Train the Autoencoder

    Open Auto_Encoder.ipynb.

    The training notebook should:

    1. Load grayscale image data, for example MNIST.
    2. Normalize images to the range [0, 1].
    3. Add artificial noise to the clean images.
    4. Train the autoencoder to map:
    noisy image → clean image

    Example training objective:

    model.fit(
    x_train_noisy,
    x_train_clean,
    validation_data=(x_test_noisy, x_test_clean),
    epochs=...,
    batch_size=...
)

    After training, test the model in Python first. Compare:

    clean image
noisy image
denoised image from Keras

    Only continue to Noodle deployment after the Keras model gives reasonable denoising results.

    6. Step 2 — Export Weights for Noodle

    For this autoencoder, use exporter_model(), not only model.get_weights().

    This is important because Conv2D and Conv2DTranspose are both 4D tensors, but their Keras layouts are different. The layer-aware exporter correctly handles both.

    Example:

    from model_exporter import exporter_model

exporter_model(model, "exported_weights")

    The exporter will generate files similar to:

    exported_weights/
├── w01.h
├── b01.h
├── w02.h
├── b02.h
├── w03.h
├── b03.h
├── w04.h
└── b04.h

    Copy these files into the Arduino/PlatformIO project folder.

    7. Step 3 — Prepare the Microcontroller Project

    The firmware must include:

    #include "noodle.h"
#include "noodle_serial.h"

#include "w01.h"
#include "b01.h"
#include "w02.h"
#include "b02.h"
#include "w03.h"
#include "b03.h"
#include "w04.h"
#include "b04.h"

    The image size is fixed to 28 × 28:

    static constexpr uint16_t IMG_W = 28;
static constexpr uint16_t IMG_H = 28;
static constexpr uint16_t IMG_SIZE = IMG_W * IMG_H;

    The firmware receives IMG_SIZE = 784 bytes from Python, converts the bytes to float values in [0, 1], runs Noodle inference, converts the output back to bytes, and sends the denoised image back.

    8. Step 4 — Understand the Noodle Forward Pass

    The autoencoder forward pass has four layers:

    W = noodle_conv_float(X,  C_IN, C1, Z1, W, E1, no_pool, NULL);
W = noodle_conv_float(Z1, C1, C2, Z2, W, E2, no_pool, NULL);

W = noodle_conv_transpose_float(Z2, C2, C3, Z3, W, D1, NULL);
W = noodle_conv_transpose_float(Z3, C3, C_OUT, Y, W, D2, NULL);

noodle_sigmoid(Y, IMG_SIZE);

    The encoder uses normal convolution:

    E1.OP = 0;
E2.OP = 0;

    The decoder uses transposed convolution:

    D1.OP = 1;
D2.OP = 1;

    The last layer uses ACT_NONE in the ConvMem structure, then applies sigmoid manually:

    D2.act = ACT_NONE;
noodle_sigmoid(Y, IMG_SIZE);

    This matches the Keras decoder output activation.

    9. Step 5 — Preparing Memory Variables with Ping-Pong Buffers

    For the first student version, use memory-to-memory inference. This means all tensors are stored in RAM, not on an SD card or external file.

    However, we do not need to allocate one permanent buffer for every layer output.

    The naive view is:

    X  → enc1 → Z1
Z1 → enc2 → Z2
Z2 → dec1 → Z3
Z3 → dec2 → Y

    This suggests five buffers:

    X, Z1, Z2, Z3, Y

    But during straight-line inference, only two tensors are needed at a time:

    current input tensor
next output tensor

    So we can use ping-pong buffers:

    BUF_A → layer → BUF_B
BUF_B → layer → BUF_A
BUF_A → layer → BUF_B
BUF_B → layer → BUF_A

    This is the recommended memory-to-memory version for students.

    9.1 Tensor sizes

    The autoencoder uses the following tensor shapes:

    Tensor Meaning Shape Number of floats
    X input image 1 × 28 × 28 784
    Z1 output of encoder 1 16 × 14 × 14 3136
    Z2 output of encoder 2 32 × 7 × 7 1568
    Z3 output of decoder 1 16 × 14 × 14 3136
    Y final denoised image 1 × 28 × 28 784

    The largest tensor is:

    16 × 14 × 14 = 3136 floats

    Therefore, each ping-pong buffer must be able to store at least 3136 floats.

    BUF_A = 3136 floats
BUF_B = 3136 floats

    Memory requirement:

    2 × 3136 floats × 4 bytes = 25088 bytes
                         ≈ 24.5 KB

    This is smaller than allocating all five tensors permanently:

    X  = 1  × 28 × 28 = 784 floats
Z1 = 16 × 14 × 14 = 3136 floats
Z2 = 32 × 7  × 7  = 1568 floats
Z3 = 16 × 14 × 14 = 3136 floats
Y  = 1  × 28 × 28 = 784 floats

total = 9408 floats
      = 9408 × 4 bytes
      = 37632 bytes
      ≈ 36.75 KB

    So the ping-pong version saves:

    37632 - 25088 = 12544 bytes
              ≈ 12.25 KB

    9.2 Declare model dimensions

    Declare the tensor dimensions clearly. This helps students connect the C++ firmware to the Keras model summary.

    static constexpr uint16_t W_IN  = 28;
static constexpr uint16_t C_IN  = 1;

static constexpr uint16_t W1    = 14;
static constexpr uint16_t C1    = 16;

static constexpr uint16_t W2    = 7;
static constexpr uint16_t C2    = 32;

static constexpr uint16_t W3    = 14;
static constexpr uint16_t C3    = 16;

static constexpr uint16_t W_OUT = 28;
static constexpr uint16_t C_OUT = 1;

    The important rule is:

    number of floats = C × W × W

    For this example, all feature maps are square.

    9.3 Declare ping-pong buffers

    Instead of declaring five tensor pointers:

    static float *X  = nullptr;
static float *Z1 = nullptr;
static float *Z2 = nullptr;
static float *Z3 = nullptr;
static float *Y  = nullptr;

    declare only two large working buffers:

    static constexpr size_t PINGPONG_FLOATS = 16 * 14 * 14;

static float *BUF_A = nullptr;
static float *BUF_B = nullptr;

    These two buffers will take turns storing the current layer input and the next layer output.

    9.4 Allocate ping-pong memory and Noodle temporary memory

    Use one helper function so the code works on ESP32 boards with or without PSRAM:

    static float *alloc_float_buffer(size_t n) {
#if defined(ARDUINO_ARCH_ESP32)
  float *p = (float *)ps_malloc(n * sizeof(float));
  if (p) return p;
#endif
  return (float *)malloc(n * sizeof(float));
}

    Then allocate the two ping-pong buffers and the Noodle temporary buffer.

    For this student guide, use the beginner-safe TEMP2 size:

    TEMP2 = 28 × 28 floats

    This is larger than the exact minimum, but easier to explain.

    static constexpr size_t PINGPONG_FLOATS = 16 * 14 * 14;
static constexpr size_t TEMP2_FLOATS    = 28 * 28;

static float *BUF_A = nullptr;
static float *BUF_B = nullptr;
static float *TEMP2 = nullptr;

static void alloc_buffers() {
  BUF_A = alloc_float_buffer(PINGPONG_FLOATS);
  BUF_B = alloc_float_buffer(PINGPONG_FLOATS);
  TEMP2 = alloc_float_buffer(TEMP2_FLOATS);

  if (!BUF_A || !BUF_B || !TEMP2) {
    Serial.println(F("ERROR allocation failed"));
    while (true) delay(1000);
  }

  // Required before using memory-to-memory Conv2D.
  noodle_setup_temp_buffers(TEMP2);

  Serial.println(F("Ping-pong and temp buffers allocated"));
}

    This is the main memory preparation step.

    The important idea is:

    BUF_A and BUF_B are for layer tensors.
TEMP2 is for internal Conv2D accumulation.

    9.5 Input and output byte buffers

    The serial protocol sends images as bytes, not floats. Therefore, the firmware also needs byte buffers:

    static uint8_t RX_BYTES[IMG_SIZE];
static uint8_t TX_BYTES[IMG_SIZE];

    RX_BYTES stores the incoming noisy image from Python.

    TX_BYTES stores the outgoing denoised image sent back to Python.

    The conversion is:

    Python uint8 image → RX_BYTES → BUF_A as float [0,1]
final output float → TX_BYTES → Python uint8 image

    9.6 Convert input bytes to floats

    Before inference, convert the received image to floating-point values in BUF_A:

    static void bytes_to_float_image_0_1(const uint8_t *src, float *dst, size_t n) {
  const float inv = 1.0f / 255.0f;
  for (size_t i = 0; i < n; i++) {
    dst[i] = (float)src[i] * inv;
  }
}

    This matches the Python/Keras normalization.

    9.7 Convert output floats to bytes

    After inference, convert the denoised image back to uint8_t:

    static void float_image_to_bytes_0_255(const float *src, uint8_t *dst, size_t n) {
  for (size_t i = 0; i < n; i++) {
    float v = src[i];

    if (v < 0.0f) v = 0.0f;
    if (v > 1.0f) v = 1.0f;

    int q = (int)(255.0f * v + 0.5f);

    if (q < 0) q = 0;
    if (q > 255) q = 255;

    dst[i] = (uint8_t)q;
  }
}

    This makes the result easy to plot in Python.

    9.8 Ping-pong layer mapping

    The forward pass becomes:

    static uint16_t run_autoencoder_forward_pingpong() {
  ConvMem E1;
  E1.K = 3; E1.P = 65535; E1.S = 2; E1.OP = 0;
  E1.weight = w01; E1.bias = b01; E1.act = ACT_RELU;

  ConvMem E2;
  E2.K = 3; E2.P = 65535; E2.S = 2; E2.OP = 0;
  E2.weight = w02; E2.bias = b02; E2.act = ACT_RELU;

  ConvMem D1;
  D1.K = 3; D1.P = 65535; D1.S = 2; D1.OP = 1;
  D1.weight = w03; D1.bias = b03; D1.act = ACT_RELU;

  ConvMem D2;
  D2.K = 3; D2.P = 65535; D2.S = 2; D2.OP = 1;
  D2.weight = w04; D2.bias = b04;
  D2.act = ACT_NONE;

  Pool no_pool;
  no_pool.M = 1;
  no_pool.T = 1;

  float *A = BUF_A;
  float *B = BUF_B;

  uint16_t W = W_IN;

  // X is in A, Z1 will be in B.
  W = noodle_conv_float(A, C_IN, C1, B, W, E1, no_pool, NULL);

  // Z1 is in B, Z2 will be in A.
  W = noodle_conv_float(B, C1, C2, A, W, E2, no_pool, NULL);

  // Z2 is in A, Z3 will be in B.
  W = noodle_conv_transpose_float(A, C2, C3, B, W, D1, NULL);

  // Z3 is in B, Y will be in A.
  W = noodle_conv_transpose_float(B, C3, C_OUT, A, W, D2, NULL);

  // Final sigmoid is applied to Y, which is now in A.
  noodle_sigmoid(A, IMG_SIZE);

  return W;
}

    The layer flow is:

    BUF_A contains X
BUF_B contains Z1
BUF_A contains Z2
BUF_B contains Z3
BUF_A contains Y

    So after inference, the final output image is in BUF_A.

    9.9 Processing one image

    The complete image-processing step becomes:

    static void process_one_image() {
  // RX_BYTES → BUF_A
  bytes_to_float_image_0_1(RX_BYTES, BUF_A, IMG_SIZE);

  uint32_t t0 = micros();
  uint16_t W_final = run_autoencoder_forward_pingpong();
  uint32_t dt = micros() - t0;

  if (W_final != W_OUT) {
    Serial.printf("ERR_BAD_W %u\n", W_final);
    NoodleSerial::print_ready();
    return;
  }

  // BUF_A contains the final denoised output.
  float_image_to_bytes_0_255(BUF_A, TX_BYTES, IMG_SIZE);

  NoodleSerial::send_output_image(TX_BYTES, IMG_SIZE, dt);
}

    9.10 Why ping-pong works here

    Ping-pong buffering works because this autoencoder is a straight-line network:

    layer 1 → layer 2 → layer 3 → layer 4

    Each layer only needs the output of the previous layer.

    After Z1 has been used to compute Z2, we no longer need Z1.

    After Z2 has been used to compute Z3, we no longer need Z2.

    This would not be enough for all networks. Models with skip connections, residual connections, concatenation, or branches may need to keep older tensors alive for longer.

    For this denoising autoencoder, ping-pong buffering is safe.

    9.11 Important: ping-pong buffers are not temporary buffers

    The ping-pong buffers store layer tensors:

    X, Z1, Z2, Z3, Y

    Noodle temporary buffers store internal operator scratch memory.

    These are different concepts.

    A simple way to remember it is:

    BUF_A and BUF_B hold tensors between layers.
TEMP2 is scratch paper used while one Conv2D layer is being computed.

    So students should not treat TEMP2 as another layer output buffer. It is only temporary workspace.

    A slightly more technical version is:

    Ping-pong buffers store data that must survive until the next layer.
TEMP2 stores temporary values that are only needed while the current operator is running.

    So the complete memory idea is:

    BUF_A / BUF_B  = layer tensor storage
TEMP2          = internal Conv2D scratch storage
RX_BYTES       = serial input bytes
TX_BYTES       = serial output bytes

    The next section explains TEMP2.

    9.12 Memory preparation checklist

    Before running inference, students should check:

    Item Expected value
    Input image size 28 × 28
    Input byte count 784 bytes
    Largest activation tensor 16 × 14 × 14
    Ping-pong buffer size 3136 floats each
    Number of ping-pong buffers 2
    Ping-pong tensor memory 25088 bytes
    Input byte buffer 784 bytes
    Output byte buffer 784 bytes
    Input normalization divide by 255
    Output conversion clip to [0,1], multiply by 255
    Final output location BUF_A

    For this beginner guide, the goal is not yet to use SD card storage. The goal is to understand how tensors move through memory.

    After students understand this memory-to-memory ping-pong version, they can move to the more advanced Noodle style where activations, parameters, or both are stored externally.

    10. Step 6 — Temporary Buffers

    Noodle uses temporary buffers as reusable scratch memory. These buffers are not model weights and they are not permanent layer outputs. They are only working space used inside some operators.

    In this guide, think of TEMP2 as scratch paper: it is used briefly during Conv2D and then reused. It does not store a named layer output such as Z1, Z2, Z3, or Y.

    For this image denoising case, the important rule is:

    Conv2D needs temp_buff2.
Conv2DTranspose in this memory-to-memory case does not need temp_buff1/temp_buff2.

    The current firmware uses memory-to-memory convolution:

    W = noodle_conv_float(X,  C_IN, C1, Z1, W, E1, no_pool, NULL);
W = noodle_conv_float(Z1, C1, C2, Z2, W, E2, no_pool, NULL);

    Even though both input and output tensors are already in memory, noodle_conv_float() still uses temp_buff2 as an accumulation buffer for one output channel plane.

    10.1 What does temp_buff2 store?

    During Conv2D, Noodle does not immediately write the final output tensor. For each output channel, it first accumulates:

    input channel 0 contribution
+ input channel 1 contribution
+ ...
+ input channel N contribution

    into one temporary output plane.

    After accumulation, Noodle applies:

    bias → activation → pooling

    Then the result is copied into the real output tensor.

    So temp_buff2 must be large enough to hold one pre-pooling output plane:

    temp_buff2 size = Vconv × Vconv floats

    where Vconv is the Conv2D output width before pooling.

    For SAME-style stride-2 convolution in this example:

    enc1: input 28×28, stride 2 → output 14×14
enc2: input 14×14, stride 2 → output 7×7

    Therefore:

    enc1 temp_buff2 = 14 × 14 = 196 floats
enc2 temp_buff2 = 7 × 7 = 49 floats

    The largest requirement is:

    max temp_buff2 = 196 floats
               = 196 × 4 bytes
               = 784 bytes

    10.2 Recommended student version

    For clarity, students can allocate a slightly larger scratch buffer:

    static constexpr uint16_t TEMP2_FLOATS = IMG_W * IMG_W;
static float *TEMP2 = nullptr;

    This gives:

    TEMP2_FLOATS = 28 × 28 = 784 floats
TEMP2 memory = 784 × 4 = 3136 bytes

    This is larger than necessary, but easy to understand because it matches one full image plane.

    In the final student code, TEMP2 is allocated together with BUF_A and BUF_B inside alloc_buffers().

    The critical line is:

    noodle_setup_temp_buffers(TEMP2);

    This line registers the scratch buffer with Noodle before any noodle_conv_float() call runs.

    10.3 More memory-efficient version

    After students understand the concept, they can use the exact minimum:

    static constexpr uint16_t TEMP2_FLOATS = 14 * 14;
static float *TEMP2 = nullptr;

    Then:

    TEMP2 = alloc_float_buffer(TEMP2_FLOATS);
noodle_setup_temp_buffers(TEMP2);

    This only uses:

    196 floats × 4 bytes = 784 bytes

    10.4 Do we need temp_buff1?

    For this exact firmware, no.

    temp_buff1 is mainly needed when Noodle must stream input planes from a file or other external storage. In this denoising firmware, the input and intermediate tensors are already in memory:

    X, Z1, Z2, Z3, Y

    So the current memory-to-memory Conv2D path only needs temp_buff2.

    If later students move activations to SD card, then temp_buff1 becomes necessary. In that case, size it for the largest input plane:

    max temp_buff1 = 28 × 28 floats
               = 784 floats
               = 3136 bytes

    and register both buffers:

    noodle_setup_temp_buffers(TEMP1, TEMP2);

    10.5 Temporary buffer checklist

    Before running inference, students should answer:

    Question Answer for this denoising case
    Are we using Conv2D? Yes
    Does Conv2D need temp_buff2? Yes
    Largest Conv2D pre-pooling plane? 14 × 14
    Minimum temp_buff2 size? 196 floats
    Beginner-safe temp_buff2 size? 784 floats
    Do we need temp_buff1? No, not for memory-to-memory tensors
    Do Conv2DTranspose layers need shared temp buffers here? No
    Are BUF_A and BUF_B the same as TEMP2? No, they are layer tensor buffers

    A missing temp buffer usually causes noodle_conv_float() to return 0, which then leads to a wrong layer width and finally:

    ERR_BAD_W 0

    11. Step 7 — Serial Protocol

    The microcontroller and Python communicate using a simple serial protocol.

    Microcontroller startup

    After boot and buffer allocation, the microcontroller prints:

    READY

    Python sends one image

    Python sends the ASCII header:

    IMG

    Then the microcontroller replies:

    RDYIMG

    Python sends the image bytes in chunks. The current chunk size is 64 bytes.

    After each received chunk, the microcontroller replies:

    ACK

    Microcontroller sends the denoised image

    After inference, the microcontroller sends:

    OUT <inference_time_us>

    Then it sends 784 output bytes.

    Finally, it prints:

    READY

    This means the microcontroller is ready for the next image.

    12. Step 8 — Python Host Test

    The Python host should:

    1. Open the serial port.
    2. Wait until it receives READY.
    3. Select one noisy 28 × 28 image.
    4. Convert it to uint8:
    img_u8 = np.clip(noisy_img * 255, 0, 255).astype(np.uint8)
    1. Send the header:
    ser.write(b"IMG")
    1. Wait for RDYIMG.
    2. Send the 784 image bytes in 64-byte chunks.
    3. Wait for ACK after every chunk.
    4. Wait for the OUT <time_us> line.
    5. Read 784 output bytes.
    6. Reshape the result:
    out = np.frombuffer(raw, dtype=np.uint8).reshape(28, 28)
    1. Plot the result:
    clean image | noisy image | Noodle denoised image

    A successful serial log should look like this:

    BOOT OK

READY
RDYIMG
ACK
ACK
...
OUT 123456
READY

    The exact inference time depends on the board.

    14. Common Problems

    Problem: Python never receives READY

    Reset the board or reopen the serial port. The firmware prints READY repeatedly if no valid IMG header is received, so the protocol should recover automatically.

    Problem: ERR_TIMEOUT

    Python did not send the image bytes fast enough or the serial port was interrupted. Check the baud rate and make sure the host sends exactly 784 bytes.

    Problem: ERR_BAD_W

    The final output width is not 28. Check that the model architecture in Keras matches the constants in main.cpp.

    Also check that the padding mode matches the Keras model. For this denoising autoencoder, all four layers should use SAME padding:

    E1.P = 65535;
E2.P = 65535;
D1.P = 65535;
D2.P = 65535;

    If one layer accidentally uses P = 0, the width sequence may no longer be:

    28 → 14 → 7 → 14 → 28

    Also check that TEMP2 was allocated and registered:

    TEMP2 = alloc_float_buffer(TEMP2_FLOATS);
noodle_setup_temp_buffers(TEMP2);

    If TEMP2 is missing, the Conv2D path may fail and return width 0.

    Problem: Output image is black or white

    Check these items:

    1. Input normalization must be [0, 1] during training.
    2. Python must send uint8 values from 0 to 255.
    3. Firmware must divide by 255 before inference.
    4. Firmware must apply sigmoid after the final decoder layer.
    5. Output must be clipped to [0, 1] before converting back to bytes.

    Problem: Output shape is correct but image quality is poor

    Check that:

    1. The exported wXX.h and bXX.h files came from the latest trained model.
    2. The layer order in the firmware matches the Keras model.
    3. Conv2DTranspose weights were exported with exporter_model().
    4. The training noise level is similar to the test noise level.

    16. Final Benchmark: Running the Same Autoencoder with TFLite Micro

    After students successfully run the image denoising model with Noodle, they can benchmark the same model using TensorFlow Lite Micro.

    The goal is not to replace Noodle. The goal is to compare two embedded inference styles:

    Framework Main idea
    Noodle Explicit layer-by-layer inference with visible memory planning
    TFLite Micro Interpreter-based inference using a .tflite model and tensor arena

    This benchmark helps students understand the trade-off between manual control and framework automation.

    16.1 Export the Keras model to .tflite

    After training the autoencoder, convert it to a float TFLite model:

    import tensorflow as tf
import os

save_dir = "/content/drive/MyDrive/NOODLE/autoencoder"

converter = tf.lite.TFLiteConverter.from_keras_model(autoencoder)

# Keep this benchmark simple first: use float32, not int8.
tflite_model = converter.convert()

tflite_path = os.path.join(save_dir, "mnist_denoising_autoencoder_float.tflite")
with open(tflite_path, "wb") as f:
    f.write(tflite_model)

print("Saved:", tflite_path)
print("TFLite size:", len(tflite_model), "bytes")

    For the first benchmark, use float32 TFLite. Quantized int8 TFLite can be added later after the float version works.

    16.2 Convert .tflite to a C header

    The ESP32 Arduino sketch needs the model as a C array.

    In Linux or Google Colab, use:

    xxd -i mnist_denoising_autoencoder_float.tflite > model_data.h

    Then edit the generated header names, or use this Python version to generate cleaner names:

    from pathlib import Path

tflite_path = Path("mnist_denoising_autoencoder_float.tflite")
data = tflite_path.read_bytes()

with open("model_data.h", "w") as f:
    f.write("#pragma once\n\n")
    f.write("#include <stdint.h>\n\n")
    f.write(f"const unsigned int g_model_len = {len(data)};\n")
    f.write("const unsigned char g_model[] = {\n")

    for i in range(0, len(data), 12):
        row = data[i:i+12]
        f.write("  " + ", ".join(f"0x{b:02x}" for b in row) + ",\n")

    f.write("};\n")

    Copy model_data.h into the ESP32 Arduino project folder.

    16.3 ESP32 TFLite Micro project files

    The TFLite Micro benchmark project should contain:

    esp32_tflm_denoising/
├── main_tflm_denoising_benchmark.cpp
├── model_data.h
├── noodle_serial.h
└── noodle_serial.cpp

    This benchmark intentionally reuses noodle_serial.h and noodle_serial.cpp.

    That means the serial protocol remains the same:

    READY → IMG → RDYIMG → ACK → OUT <time_us> → output image bytes

    So the same Python host script used for Noodle can also be used for TFLite Micro.

    16.4 Tensor arena

    TFLite Micro does not ask students to manually allocate BUF_A, BUF_B, and TEMP2.

    Instead, it uses one large memory region called the tensor arena.

    A simple explanation is:

    Noodle:
  BUF_A / BUF_B = layer tensor storage
  TEMP2         = operator scratch storage

TFLite Micro:
  tensor_arena  = framework-managed storage for tensors and scratch memory

    So for TFLite Micro, students allocate:

    constexpr size_t TENSOR_ARENA_SIZE = 512 * 1024;
static uint8_t *tensor_arena = nullptr;

    Then the interpreter decides how to use that arena internally.

    If allocation fails, increase available memory by using an ESP32 board with PSRAM.

    If AllocateTensors() fails, increase TENSOR_ARENA_SIZE.

    16.5 Required TFLite Micro operators

    This autoencoder usually needs these operators:

    CONV_2D
TRANSPOSE_CONV
LOGISTIC

    Depending on the converter version, ReLU may be fused into Conv2D/TransposeConv. If it is not fused, the model may also need a standalone ReLU operator.

    For the beginner benchmark, the provided ESP32 code uses AllOpsResolver to avoid missing-operator problems.

    After the benchmark works, students can replace AllOpsResolver with a smaller MicroMutableOpResolver to reduce flash usage.

    16.6 Benchmark comparison table

    Students should record:

    Metric Noodle TFLite Micro
    Model format .h weight arrays .tflite C array
    Input size 784 bytes 784 bytes
    Output size 784 bytes 784 bytes
    Inference time OUT <time_us> OUT <time_us>
    RAM strategy ping-pong + TEMP2 tensor arena
    Memory control explicit framework-managed
    Ease of debugging layer shapes high lower
    Deployment convenience manual easier once model converts

    This comparison is useful because it shows the educational difference between Noodle and TFLite Micro.

    Noodle makes the memory movement visible.

    TFLite Micro hides most memory movement inside the interpreter.

    17. Main Learning Points

    After completing this exercise, students should understand that:

    1. A convolutional autoencoder can perform image denoising.
    2. Noodle can run convolution and transposed convolution on a microcontroller.
    3. Keras tensor layout and Noodle tensor layout are different.
    4. Weight export is a critical deployment step.
    5. Serial communication is needed to move input and output images between Python and the microcontroller.
    6. Embedded inference is not only about model accuracy, but also memory, data layout, and communication protocol.
